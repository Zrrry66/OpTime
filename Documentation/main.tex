\section{Introduction}
Timetable scheduling is a hard problem. The number of possible combinations of courses, rooms, time slots, and lecturers grows very fast, and finding a schedule that satisfies everyone's constraints manually is both slow and error-prone.

The project is a web-based tool to automate this process for the Faculty of Media, and it was developed by three sub-teams:

\begin{itemize}
    \item \textbf{Frontend (UI/UX)}: Built the user interface for managing lecturers, courses, and rooms, and for visualizing the generated timetable.
    \item \textbf{Gurobi}: Used Integer Linear Programming to generate an initial valid schedule that satisfies all hard constraints.
     \item \textbf{Simulated Annealing}: Applied a metaheuristic algorithm to improve the current schedule by reducing soft constraint violations.
\end{itemize}

The three parts work together in a user-driven process: the frontend collects all input data and sends it to the Gurobi solver to create a base schedule. Users can then manually adjust the results through the interface. After that, the Simulated Annealing module takes that version and improves it further. The final result is shown on the frontend, ready to be reviewed and exported. This document covers each component in detail.

Before we started building the OpTime System, our group analyzed 13 research papers to study the principles of various optimization algorithms and their practical applications in university timetabling. \subsection{Problem Complexity}
The literature defines university timetabling as an NP-complete or NP-hard problem, meaning no efficient algorithm can find a perfect solution every time as the data size grows \cite{burke1997automated, schaerf1999survey}. To address this issue, exhaustive enumeration of all feasible configurations in search of an optimal one is avoided. Instead, a suboptimal solution is sought. Several approaches exist for this purpose, including, among others\cite{alghamdi2020review}:

\begin{itemize}
    \item The use of off-the-shelf solvers (engines) based on integer programming, where the problem is formulated by specifying constraints interpretable by the model in such a way that the solution to the formulated problem corresponds to a feasible solution of the original problem.
    
    \item Algorithms such as simulated annealing, implemented either using existing frameworks or developed from scratch, which iteratively select improved configurations from a set of modifiable candidates, exclude certain configurations according to problem-specific constraints, and evaluate admissible configurations using an objective function.

    \item The use of genetic algorithms, which mimic key aspects of the evolutionary process. In this approach, new configurations are generated from existing ones, making it conceptually related to the method described above, while also incorporating mechanisms for repairing infeasible solutions. By imposing limits on the number of iterations or steps in these and the previously mentioned algorithms, it is possible to obtain a suboptimal yet satisfactory solution within a reasonable time frame.\cite{Kristiadi2019}
    
    \item Hybrid approaches that combine multiple algorithms, distributing tasks between them in order to avoid evaluating all possible configurations, while compensating for this by enabling transitions from inferior to superior configurations.\cite{Yunita2018}


    
\end{itemize}


\subsection{Algorithm Mechanisims and Applications}
Our review focused on the mechanics of several optimization methods:
\begin{itemize}
    \item \textbf{Genetic Algorithms (GA)}: These use a biological evolution approach involving selection, crossover, and mutation. We found that Violated Directed Mutation (VDM) is particularly effective because it specifically targets and fixes genes that violate constraints rather than making random changes \cite{Kristiadi2019}.
    \item \textbf{Simulated Annealing (SA)}: This algorithm keeps track of a current solution and occasionally accepts ``worse'' moves as the system ``temperature'' decreases. This mechanism helps the solver escape local optima to find a better overall schedule.
    \item \textbf{Ant Colony Optimization (ACO) \& PSO}: ACO mimics ant behavior by using pheromones to find the best paths to a solution. When hybridized with Particle Swarm Optimization (PSO), the system can automatically tune its own parameters, making the search process more stable and successful \cite{Yunita2018}.
    \item \textbf{Fuzzy Logic}: Some models use fuzzy logic to handle uncertain human preferences, which helps in balancing conflicting soft constraints \cite{robinson2025adaptive}.
\end{itemize}

These insights led us to adopt a dual-solver architecture. While Integer Programming (IP) models can provide near-optimal solutions, in practice such solutions may fail to meet all practical requirements. This motivates the need for a mechanism to modify the obtained solution while preserving its feasibility and overall quality. The first component of the approach, due to its deterministic nature and lack of stochasticity, is not capable of providing such flexibility, thereby necessitating a complementary method.

We addressed these performance constraints by adopting a dual-solver architecture. We use Simulated Annealing for the drafting and real-time editing phases, as it provides the immediate feedback necessary for an interactive tool. This hybrid setup allows OpTime to stay fast and responsive at scale, while ensuring all results remain grounded in the formal logic of the IP-based model \cite{strichman2017near}.

\subsection{Technical Implementation}

\subsubsection{Front-End Architecture}

The front-end system is implemented as a \textbf{modular single-page application (SPA)} that integrates multiple functional views within a unified interface. The architecture is designed to separate concerns between data management, interaction logic, and visualization, ensuring both scalability and maintainability.

\begin{itemize}
    \item \textbf{Modular Structure}: The system is decomposed into independent feature modules (e.g., CourseModule, LecturerModule, RoomModule, TimetableModule), each responsible for a specific domain. These modules are initialized and coordinated through a central entry point, enabling loose coupling and extensibility.
    
    \item \textbf{Centralized Data Management}: A singleton \textbf{DataManager} maintains a consistent global state across all modules. This ensures that updates in one component (e.g., editing a course or lecturer) are immediately reflected throughout the system.
    
    \item \textbf{Dynamic View Switching}: The interface organizes content into multiple views (Lecturers, Courses, Rooms, Timetable), which are dynamically activated without page reload. This allows seamless navigation while preserving user context and interaction state.
    
    \item \textbf{Reusable UI Components}: HTML partials are loaded dynamically at runtime, enabling flexible composition of interface elements such as modals, drawers, and tool panels. This reduces redundancy and simplifies future interface extensions.
    
    \item \textbf{Structured Layout Design}: The interface combines a persistent sidebar for global actions (navigation, generation, data management) with a main content area for task-specific interaction. This layout supports efficient workflows and minimizes user disorientation.
    
\end{itemize}
This modular architecture improves maintainability and scalability, while ensuring a clear separation of concerns.
It also supports a responsive and predictable interface, which enhances usability and user experience.



\subsubsection{The Dynamic Timetable Engine}

The core functionality of the system is implemented through a \textbf{dynamic timetable engine} that integrates algorithmic scheduling results with interactive front-end manipulation. This component transforms backend optimization outputs into a structured, user-controllable visual representation.

\begin{itemize}
    \item \textbf{State-Driven Rendering}: The timetable is generated from structured assignment data returned by the backend solver. Each entry is mapped to a visual "Course Card," whose position is determined by its assigned day and time-slot index. The system maintains an internal state representation of all assignments, ensuring consistency between data and visualization.
    
    \item \textbf{Interactive Visualization Layer}: The timetable is rendered as a grid aligned with temporal structure (days and time slots), allowing users to intuitively interpret scheduling outcomes. Each grid cell dynamically aggregates multiple assignments, supporting real-time updates and filtering (e.g., by faculty or semester).
    
    \item \textbf{Real-Time Conflict Visualization}: The system integrates a validation layer that highlights constraint violations directly within the interface. Critical conflicts (e.g., lecturer overlap, room capacity issues) are encoded using red visual indicators, while softer constraint violations are represented using warning colors. This enables immediate identification of problematic assignments.
    
    \item \textbf{Drag-and-Drop Interaction}: Users can manually adjust scheduling results by dragging course cards across the grid. The system provides visual hints about slot suitability (available, preferred, or unavailable) and prevents invalid placements. Upon each drop action, the system updates the internal state and triggers validation to maintain constraint consistency.
    
    \item \textbf{Iterative Update Mechanism}: The engine supports continuous refinement by allowing users to regenerate or improve schedules based on current assignments. User modifications are preserved and re-integrated into subsequent optimization steps, enabling an interactive exploration of alternative scheduling solutions.
\end{itemize}

\subsubsection{Asynchronous Operation Handling}

To ensure a responsive and uninterrupted user experience, the system employs \textbf{asynchronous operation handling} for data loading, scheduling computation, and system interaction.

\begin{itemize}
    \item \textbf{Asynchronous Initialization}: During application startup, HTML partials and initial data are loaded using asynchronous operations. The system waits for all components to be successfully initialized before rendering the interface, ensuring consistency while avoiding blocking the main thread.
    
    \item \textbf{Promise-Based Backend Communication}: The front-end communicates with the scheduling solver through \texttt{fetch}-based API calls. This allows timetable generation and improvement processes to run in the background while the user interface remains interactive.
    
    \item \textbf{Loading State Feedback}: During computationally intensive operations (e.g., schedule generation or improvement), the system displays a loading overlay with status messages (e.g., "Optimizing schedule" or "Improving schedule"). This provides clear feedback about system status and reduces user uncertainty.
    
    \item \textbf{Non-Blocking Interaction Model}: User interactions such as editing assignments, dragging course cards, or switching views are handled without interrupting the application flow. Updates are applied immediately to the local state, while validation and backend processing occur asynchronously.
    
    \item \textbf{Selective Rendering}: The system conditionally renders content based on the active view, avoiding unnecessary updates and improving performance. This ensures that only relevant components are updated in response to user actions.
\end{itemize}

\subsubsection{Bridge Between User and Algorithm}

The system establishes a \textbf{bidirectional interaction bridge} between user input and the scheduling algorithm, transforming abstract optimization constraints into an interactive and controllable interface.

\begin{itemize}

These selections are directly translated into structured constraint data used by the optimization engine.
    
    \item \textbf{Unified Data Representation}: All user-defined inputs (courses, lecturers, rooms, and time constraints) are collected and managed through a centralized data layer. This ensures consistency between user actions and algorithmic computation.
    
    \item \textbf{Bidirectional Interaction}: Users can both initiate automated scheduling (via "Generate" or "Improve") and manually adjust results (e.g., drag-and-drop or edit dialogs). Each interaction updates the system state and can be re-evaluated by the backend solver.
    
    \item \textbf{Human-in-the-Loop Optimization}: The system supports an iterative cycle in which users refine schedules based on algorithmic feedback. User modifications are preserved and re-submitted to the solver, enabling continuous improvement of scheduling outcomes.

\subsubsection{Intelligent Conflict Reporting}

To support user understanding and decision-making, the system provides \textbf{multi-level conflict detection and feedback} mechanisms.

\begin{itemize}
    \item \textbf{Constraint Classification}: Detected violations are categorized into two types:
        \begin{itemize}

            \item \textbf{Preference Warnings (Soft Constraints)}: Non-critical deviations from preferred conditions (e.g., suboptimal time slots or room assignments).

    \item \textbf{Real-Time Validation}: After each user interaction (e.g., manual edits or drag-and-drop operations), the system triggers validation processes to ensure that the current schedule remains consistent with defined constraints.

\subsubsection{Optimization Workflow \& Persistence}

The system supports an \textbf{iterative and mixed-initiative workflow} that combines automated optimization with user-driven refinement.

\begin{itemize}
    \item \textbf{Iterative Scheduling Process}: Users first generate an initial timetable, then refine it through manual adjustments or by invoking the "Improve" function. This enables exploration of multiple feasible solutions.
    
    \item \textbf{Interactive Manipulation}: Users can directly modify scheduling results through drag-and-drop and editing interfaces. These actions immediately update the system state and trigger re-validation.
    
    \item \textbf{Version Management}: The system provides functionality for saving, loading, and managing different timetable versions. This supports comparison between alternative solutions and prevents loss of progress.
    
    \item \textbf{Data Export and Persistence}: Scheduling results can be exported (e.g., as PDF) and stored persistently. Backup and restore mechanisms ensure data safety and support long-term use.

Overall, the system transforms a complex optimization problem into an interactive, transparent, and user-controllable process, improving usability, interpretability, and user trust in algorithmic decision-making.

\section{Part II: Optimization via Gurobi}
While the Simulated Annealing module provides a fast, stochastic approach to timetable optimization, the Gurobi module takes a complementary strategy: it formulates the scheduling problem as an Integer Linear Program (ILP) and solves it to provable optimality. Integer Linear Programming is a mathematical framework in which the goal is to minimize (or maximize) a linear objective function subject to a set of linear constraints, where some or all decision variables are restricted to integer values. For binary scheduling decisions — such as whether a course is assigned to a particular room, time slot, and lecturer — this framework is a natural fit\cite{Talmor2024OptimizingAT}.

Gurobi is a state-of-the-art commercial solver for mathematical optimization problems, widely used in both industry and academic research. Compared to open-source alternatives such as GLPK or CBC, Gurobi offers significantly faster solve times on large ILP instances, a clean Python API (gurobipy), and an academic license that makes it accessible for university projects. The key trade-off between the two approaches in OpTime is as follows: Gurobi guarantees finding the globally optimal schedule but it takes much longer as the problem size grows; Simulated Annealing handles large tasks much better but it doesn't guarantee a perfect result. Together, they give users a practical choice based on their specific needs.

\subsection{Problem Formulation}
In order to solve the scheduling problem using the optimization software described above, it is necessary to construct a formal model that represents both the occurrence of a class session and the natural constraints imposed by the real-world environment in which such sessions take place. This representation must be expressed in a form that can be interpreted by a specific solver implementing a particular optimization algorithm.

The process of constructing this model can be conceptually divided into four stages, each corresponding to one of the fundamental components of the final optimization formulation. The first stage defines the decision variables, which form the foundation upon which all subsequent components of the model are built. The second stage introduces hard constraints, which establish the feasible boundaries of the problem and enforce the essential rules that must always be satisfied.The third stage incorporates soft constraints. Unlike hard constraints, these conditions represent preferences rather than strict requirements. They allow the model to be adjusted and adapted according to practical considerations and user needs.Finally, the fourth stage defines the objective function to be minimized. This function aggregates penalties associated with violations of soft constraints and provides the optimization solver with a quantitative criterion for evaluating different feasible schedules.

\subsubsection{Decision Variables}
The modeling process begins with the definition of decision variables as a multidimensional array of binary variables in Gurobi. Each unit entry in this array represents the assignment of a class to a specific time slot, taught by a particular instructor, for a given course, and in a designated room.

In Python-style notation, this can be expressed as:
\begin{verbatim}
x = m.addVars(slots, teachers_ids, courses_ids, rooms_ids, vtype=GRB.BINARY, name="x")
\end{verbatim}

Such an approach enables a natural and flexible representation of the system's constraints, as all relevant information is organized within data structures describing instructors, courses, and rooms. This facilitates the logical formulation of constraints over the entire space of possible assignments.

\section{Part III: Heuristic Optimization via Simulated Annealing (SA)}

While the Gurobi solver guarantees the best possible schedule for smaller tasks, the Simulated Annealing (SA) module is a faster alternative for larger and more complex problems. SA doesn't guarantee a ``perfect'' mathematical solution, but it quickly finds high-quality schedules—much faster than trying every possible combination. This makes it perfect for the real-time updates in the OpTime frontend.

\subsection{Simulated Annealing Algorithm}

The physical process of Annealing in metallurgy inspires Simulated Annealing. When you heat metal and then cool it slowly, the atoms have enough energy to move around and settle into a low-energy, stable configuration. If you cool it too quickly, the atoms get stuck in a suboptimal arrangement.

In optimization, we use this analogy. The ``energy'' of our system is the cost of a schedule—i.e., the number of constraint violations it has. We start with a random schedule and make small changes to it. Here is the key insight: early in the process (at high temperature), we accept many changes, even those that make the schedule slightly worse. Starting with high temperature prevents us from getting stuck in a local optimum—a schedule that's better than all its neighbors but not the best possible schedule overall. As we continue (lowering temperature), we become more selective, mainly accepting improvements.

The temperature parameter $\tau(t)$ controls this behavior. At high temperatures, we explore the solution space widely. At low temperatures, we exploit reasonable solutions by making minor refinements.

\subsubsection{Four Required Components}

To apply Simulated Annealing to the timetabling problem, we must define four components: the solution space, the cost function, the neighbor relation, and the optimization goal.

\paragraph{Solution Space}

The solution space $C$ is the set of all possible complete assignments. A solution $S \in C$ is a complete assignment of all courses in $B$ to resources in $E$.

Let $B$ be the set of courses. Let $E = R \times T \times L$ where $R$ is the set of rooms, $T$ is the set of time slots, and $L$ is the set of lecturers.

A solution $S$ assigns to each course $b \in B$ a triple consisting of room, time slot, and lecturer. For example:
$$S(\text{Photogrammetric CV}) = (\text{SR\_A}, \text{Mon-09:15-10:45}, \text{Prof. Rodehorst})$$

This means Photogrammetric CV is scheduled in room SR\_A on Monday from 9:15 to 10:45, taught by Prof. Rodehorst.

\paragraph{Cost Function}

The cost function $c : C \to \mathbb{R}$ measures how bad a schedule is. Lower cost means better schedule. We define:
$$c(S) = w_1 \cdot f_1(S) + w_2 \cdot f_2(S) + ... + w_n \cdot f_n(S)$$

where $f_i$ are individual penalty functions and $w_i$ are weights representing constraint importance.

For our implementation:
$$c(S) = 100 \cdot (\text{room conflicts}) + 100 \cdot (\text{lecturer conflicts}) + 50 \cdot (\text{workload}) + 10 \cdot (\text{gaps}) + 10 \cdot (\text{Friday})$$

For example, if a schedule has 2 room conflicts, 1 Friday afternoon class, and 3 student gaps:
$$c(S) = 100 \times 2 + 10 \times 1 + 10 \times 3 = 240$$

The penalties are classified by severity:
\begin{itemize}
    \item Hard constraints (weight = 100): Must be satisfied. Examples: no room conflicts, no lecturer conflicts
    \item Medium constraints (weight = 50): Important quality indicators. Example: maximum daily workload
    \item Soft constraints (weight = 10): Preferences. Examples: minimize gaps, avoid Friday afternoons
\end{itemize}

\paragraph{Optimization Goal}

Our optimization problem is to find a solution $S^* \in C$ such that for all $S \in C$:
$$c(S^*) \leq c(S)$$

We call such $S^*$ an optimal solution.

\paragraph{Neighbor Relation}

We define a binary relation $\sim$ on $C \times C$. For $S, S' \in C$, we say $S \sim S'$ if and only if we can obtain $S'$ from $S$ by exactly one of the following operations.

\textbf{Technique 1: Construct and destroy}

We select one course $b \in B$. Let $e$ be the current resource assigned to $b$ in $S$. We choose a new resource $e' \in E$ and create:
$$S' = (S \setminus \{(b,e)\}) \cup \{(b,e')\}$$

We remove exactly one assignment (the old one for course $b$) and add exactly one assignment (the new one for course $b$).

Example: Move Photogrammetric CV from (SR\_A, Mon-09:15, Prof. Rodehorst) to (SR\_H, Tue-09:15, Prof. Rodehorst).

\textbf{Technique 2: Swap Operation}

We select two courses $b_1, b_2 \in B$ with $b_1 \neq b_2$, and their resources $e_1, e_2 \in E$ where $(b_1,e_1) \in S$ and $(b_2,e_2) \in S$, then:
$$S' = (S \setminus \{(b_1,e_1), (b_2,e_2)\}) \cup \{(b_1,e_2), (b_2,e_1)\}$$

Example: Swap Virtual Reality at (SR\_H, Wed-13:30) with HCI Theory at (LH\_HK7, Mon-13:30).

We define the neighborhood of solution $S$ as:
$$N(S) := \{S' \in C | S \sim S'\}$$

\subsubsection{The Algorithm}

The Simulated Annealing algorithm proceeds as follows:

\begin{verbatim}
Algorithm: MinimizationSimulatedAnnealing(I)
Input: instance I
Output: solution S

1: let S be an initial solution S_0
2: let k be a constant
3: let t = 0
4: while we continue do
5:     let S' in N(S) be chosen randomly
6:     if c(S') <= c(S) then
7:         S := S'
8:     else
9:         with probability e^(-(c(S')-c(S))/(k*tau(t))) let S := S'
10:        otherwise let S unchanged
11:    end if
12:    t := t + 1
13: end while
14: Return(S)
\end{verbatim}

\subsubsection{Acceptance Probability}

When we generate a neighbor schedule $S'$ from current schedule $S$, we calculate the cost difference:
$$\Delta = c(S') - c(S)$$

If $\Delta < 0$, the neighbor is better—we always accept it. If $\Delta > 0$, the neighbor is worse—we accept it with probability:
$$P(\text{accept}) = e^{-\Delta / (k \cdot \tau(t))}$$

where $k$ is a constant and $\tau(t)$ is the current temperature.

Example: Current schedule has cost 200, neighbor has cost 250, temperature is 100, and $k=1$. Then $\Delta = 50$ and $P(\text{accept}) = e^{-0.5} \approx 0.606$ (60\% chance). This probability decreases as temperature drops.

Our implementation guarantees that hard constraints are never violated by validating each neighbor during generation, ensuring only valid neighbors are considered.
